# 25-Persona Lens Vote For The GU Source Action

## Purpose

This note stress-tests the source-action search by forcing 25 divergent lenses to make
independent top-3 recommendations, then normalizing the results into a vote.

This is not evidence for any construction. It is a prioritization device. A winning idea still
must become a GU-native carrier, computable loss channel, or executable guard before it matters.

## Ground Rules

- `SPEC.md` remains the target. The vote cannot relax the BV master equation, Noether-forced
  gamma-trace constraint, non-equivariant compensator, full BV bicomplex, or the three global
  objects.
- No target import: no `24 / 8`, no assumed K3, no fitted generation count.
- A persona votes for workstreams, not conclusions.
- A workstream is killed if it cannot produce a GU-native computable object.
- Vote weights: rank 1 = 3 points, rank 2 = 2 points, rank 3 = 1 point.

## Consensus Portfolio

The vote does not select one magic idea. It selects a three-part work program:

1. **Build GU-native minimax loss channels (B).**
   The security-budget lens becomes real only when `L_RS_BRST`, `L_boundary_index`,
   `L_theta_source`, `L_target_import`, and related losses point to concrete parent-repo objects.

2. **Use boundary finality as the carrier (C + A).**
   The strongest mathematical landing zone is a spectral-section/holonomy object connected to a
   BV-to-boundary-Dirac map. This is the best candidate interface between the traditional GU
   exposition and executable source-action tests.

3. **Install adversarial negative controls (I).**
   The repo should make it hard to accidentally count target imports, free APS knobs, acausal
   decouplings, or hand-imposed projectors as progress.

## First-Week Work Order

1. Create a loss-channel interface whose functions either compute from `lib/gu_bridge.py` objects or
   fail loudly with a named missing carrier.
2. Promote the current boundary-Dirac experiments into a named `L_boundary_symbol` or
   `L_boundary_index` channel only if the output distinguishes a real carrier from a free operator.
3. Add adversarial tests for the killed patterns in `DEAD-ENDS.md`: bulk selector replay,
   target-import normalization, assumed-K3 input, clean decoupling, and modified anti-trap commutator.
4. Keep Lean/formalization scoped to guardrails: dimensions, signatures, rank claims, dependency
   chains, and impossible target-import patterns.

## Final Recommendation

The most fruitful potential is not "cryptoeconomics" as an analogy. It is the concrete translation:

```text
source-action search = GU-native minimax loss channels
                       + boundary finality carrier
                       + adversarial anti-import oracle
```

If that triad cannot be made computable, the security-budget lens should be downgraded. If it can,
it becomes the repo's most disciplined route from speculative source-action language to executable
mathematical pressure tests.
